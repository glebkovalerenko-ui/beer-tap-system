# Realtime NFR и UX degraded-режимов

## 1) Целевые метрики realtime

### 1.1 Event-to-UI latency (сквозная задержка от события до отображения)

> Определение: время от фиксации доменного события в backend/controller до момента, когда пользователь видит актуальное состояние в UI.

| Метрика | Цель | SLO (за 5 минут) | SLI источник |
|---|---:|---:|---|
| p50 event-to-UI latency | ≤ 300 мс | 95% интервалов | backend event timestamp + UI render timestamp |
| p95 event-to-UI latency | ≤ 1200 мс | 95% интервалов | backend event timestamp + UI render timestamp |
| p99 event-to-UI latency | ≤ 2500 мс | 99% интервалов | backend event timestamp + UI render timestamp |

Допуски при деградации канала (например, fallback на polling):
- p50 ≤ 800 мс;
- p95 ≤ 3000 мс;
- p99 ≤ 6000 мс.

### 1.2 Максимальная допустимая «устарелость» статусов (status staleness)

> Определение: текущий момент времени минус `updated_at` последнего подтверждённого статуса в UI.

| Категория данных | Допустимая устарелость | Критический порог |
|---|---:|---:|
| Статус налива/дозирования | ≤ 2 с | > 5 с |
| Статус доступности кранов | ≤ 3 с | > 10 с |
| Платёжный/NFC статус | ≤ 2 с | > 5 с |
| Служебные health-статусы (controller/display-agent) | ≤ 5 с | > 15 с |

Правило окраски устарелости:
- `fresh`: до 50% допустимого порога;
- `aging`: 50–100% допустимого порога (жёлтый индикатор);
- `stale`: >100% (красный индикатор + ограничение части действий).

## 2) Realtime transport: websocket + fallback

### 2.1 Приоритет каналов
1. **Primary:** WebSocket (двунаправленный realtime-канал).
2. **Fallback #1:** Short polling (HTTP, интервал 1 с) — если WebSocket недоступен.
3. **Fallback #2:** Long polling / reduced polling (интервал 3–5 с) — при повторных сетевых ошибках.

### 2.2 Правила переключения
- Переход с WebSocket на polling после:
  - 3 подряд ошибок соединения, или
  - отсутствия heartbeat/keepalive > 10 с.
- Возврат с polling на WebSocket:
  - попытка каждые 30 с (экспоненциальный backoff до 2 мин);
  - после успешного handshake выполняется resync snapshot.

### 2.3 Требования к heartbeat и reconnect
- Heartbeat WebSocket: каждые 5 с.
- Таймаут heartbeat: 10 с.
- Reconnect backoff: 1 с → 2 с → 5 с → 10 с → 20 с → 60 с → 120 с (потолок).
- Jitter: ±20% для снижения thundering herd.

## 3) Поведение UI при частичной недоступности компонентов

## 3.1 Недоступен backend

### Блокируется
- Старт/стоп операций, требующих подтверждения backend (налив, подтверждение транзакции, изменение конфигурации).
- Любые действия, изменяющие критичное состояние.

### Доступно с предупреждением
- Просмотр последнего известного состояния.
- Локальные некритичные действия (навигация, открытие экранов справки/логов).

### UX-индикация
- Глобальный banner: «Нет связи с backend. Показаны последние данные.»
- Каждая критичная кнопка disabled + tooltip с причиной.

## 3.2 Недоступен controller

### Блокируется
- Физические операции с краном: старт/продолжение/останов налива.

### Доступно с предупреждением
- Просмотр очереди/сессий и журнала событий.
- Подготовка операции (предвыбор) без фактического запуска.

### UX-индикация
- Статус компонента controller = `offline/degraded` на видимой панели состояния.
- Для tap-card: «Устройство временно недоступно» + кнопка «Повторить».

## 3.3 Недоступен display-agent

### Блокируется
- Функции, требующие синхронного управления внешним экраном/киоском.

### Доступно с предупреждением
- Основной операторский workflow в UI.
- Отложенная отправка команд display-agent (best-effort очередь) при допустимости бизнес-логикой.

### UX-индикация
- Non-blocking предупреждение: «Проблема связи с display-agent. Основные операции доступны.»
- Отдельный счётчик/иконка неприменённых display-команд.

## 4) UX degraded-режимов

### 4.0 Resilience & Degraded Modes

Единая операционная модель для UI:

| Состояние | Что это означает | Доступно | Блокируется | Как объясняем пользователю |
|---|---|---|---|---|
| `online` | Backend и controller доступны, данные свежие | Все штатные действия, включая критичные commit-операции | Ничего сверх обычных бизнес-ограничений | «Система работает в штатном режиме.» |
| `backend degraded` | Нет подтверждённой связи с backend, но есть локальный контекст/controller heartbeat | Read-only, навигация, просмотр последнего snapshot, подготовка черновиков | Платёжные подтверждения, изменение критичных статусов, любые операции требующие backend-ack | «Связь с сервером нестабильна. Показываем последние данные; критичные действия временно недоступны.» |
| `controller-only` | Controller доступен локально, backend недоступен/частично недоступен, допускается локальная очередь | Локально-безопасные операции с постановкой в очередь (если бизнес-правила это разрешают), просмотр и локальная диагностика | Неидемпотентные/финансовые commit-операции без backend, массовые изменения конфигурации | «Работаем в локальном режиме. Действия будут отправлены после восстановления связи.» |
| `offline` | Нет связи ни с backend, ни с controller (или истёк критический heartbeat) | Только read-only по кэшу + справка/инструкции восстановления | Все действия, меняющие состояние системы | «Нет связи с оборудованием и сервером. Доступен только просмотр сохранённых данных.» |

Правило объяснения блокировок:
- возле каждой заблокированной кнопки — конкретная причина (`нет backend ack`, `controller offline`, `данные устарели`);
- в tooltip/inline-тексте всегда указывать условие разблокировки (`восстановите связь`, `обновите snapshot`, `дождитесь завершения синхронизации`);
- не использовать расплывчатые формулировки вида «Ошибка операции» без actionable причины.

### 4.1 Какие действия блокируются
Блокируются действия, которые могут привести к:
- физическому риску (запуск налива без подтверждения контроллера);
- финансовой неконсистентности (подтверждение платежа без backend);
- рассинхронизации критичных состояний (изменение статуса заказа при stale-контексте).

### 4.2 Какие доступны с предупреждением
Допускаются read-only и low-risk операции:
- просмотр карточек, журналов, истории;
- фильтрация/поиск;
- подготовка черновиков действий (без commit).

Все такие действия сопровождаются:
- inline-warning возле данных,
- отметкой времени «Обновлено N сек назад».

### 4.3 Как отображается stale data
Для каждого виджета с live-данными:
- timestamp последнего обновления (`updated_at`);
- цветовой индикатор свежести (fresh/aging/stale);
- при `stale`:
  - красный chip «Данные устарели»;
  - отключение опасных CTA;
  - предложение «Обновить сейчас» (ручной refresh).

Для системного уровня:
- общий health-widget с состоянием backend/controller/display-agent;
- фиксированный banner при деградации более 10 с.

### 4.4 Как восстанавливается консистентность после reconnect
1. Устанавливается транспорт (WebSocket или polling).
2. UI запрашивает **полный snapshot** состояния (authoritative source).
3. Локальные optimistic/queued состояния маркируются как `reconciling`.
4. Выполняется reconciliation:
   - подтверждённые backend события применяются;
   - конфликтные локальные изменения откатываются или переводятся в retry.
5. UI показывает toast: «Связь восстановлена. Данные синхронизированы.»
6. Снимаются временные блокировки действий.

Технические требования на reconciliation:
- идемпотентность операций по `operation_id`;
- дедупликация событий по `event_id`;
- монотонный `version`/`sequence` для порядка применения.

## 5) Экранные требования: «Сегодня», «Краны», «Карты»

### 5.1 Глобальный banner состояния связи
- Banner должен быть единым компонентом для трёх экранов и иметь приоритет над локальными предупреждениями.
- Banner показывает:
  - текущий режим (`online / backend degraded / controller-only / offline`);
  - длительность деградации (`N мин в degraded`);
  - источник проблемы (`backend`, `controller`, оба);
  - CTA: «Повторить подключение» и «Открыть диагностику».

Рекомендуемые тексты banner:
- `online`: «Связь стабильна. Данные актуальны.»
- `backend degraded`: «Связь с backend нестабильна. Показаны последние подтверждённые данные.»
- `controller-only`: «Локальный режим: controller доступен, backend недоступен. Операции ставятся в очередь.»
- `offline`: «Нет связи с backend и controller. Доступен только просмотр кэшированных данных.»

### 5.2 Очереди отложенных действий
- Для режимов `backend degraded` и `controller-only` (если допускается бизнес-логикой) UI обязан показывать очередь:
  - счётчик (`В очереди: N`);
  - возраст самого старого элемента (`Старейшее: 02:14`);
  - статус replay (`ожидает`, `отправляется`, `ошибка`, `требует решения`).
- На экранах:
  - **Сегодня:** агрегированный индикатор очереди смены + ссылка на детали.
  - **Краны:** очередь по каждому крану/операции (например, pending lock/unlock/service action).
  - **Карты:** очередь операций по картам/гостям, только для разрешённых безопасных действий.

### 5.3 Явные предупреждения о несинхронизированных операциях
- Каждая сущность с локальными неподтверждёнными изменениями маркируется chip-меткой `Не синхронизировано`.
- В карточке/строке сущности показывать:
  - `Локальное изменение, ждёт подтверждения backend`;
  - timestamp локального действия;
  - идентификатор операции (`operation_id`) для поддержки.
- Для потенциально рискованных случаев — отдельный warning-блок:
  - «Есть несинхронизированные операции. Финальные выводы по данным могут измениться после восстановления связи.»

### 5.4 Сценарий восстановления после reconnect (replay + conflict handling)
1. UI фиксирует событие reconnect и переводит очередь в `replaying`.
2. UI запрашивает полный snapshot-resync (authoritative source), чтобы выровнять базовое состояние перед повторной отправкой очереди.
3. Replay идёт строго в порядке `created_at`/`sequence` с идемпотентным ключом `operation_id`.
4. Для каждой операции:
   - при успехе: статус `synced`, метка `Не синхронизировано` снимается;
   - при reject из-за конфликта: статус `conflict`, операция не теряется;
   - при временной ошибке: ограниченный retry с backoff, затем `needs_manual_review`.
5. После replay выполняется reconciliation поверх актуального snapshot:
   - подтверждённые backend события применяются;
   - конфликтные локальные изменения откатываются или переводятся в retry/manual review.
6. UI показывает итог:
   - «Связь восстановлена. Синхронизировано: N.»
   - «Требуют решения: M конфликтов.» (с ссылкой на экран разбора конфликтов/инцидентов)

Минимальные правила обработки конфликтов:
- не делать silent overwrite локальных данных;
- показывать пользователю сторону конфликта (`локальная версия` vs `серверная версия`);
- предлагать явный выбор: `принять серверную`, `повторить локальную`, `отменить`.

## 6) UX-тексты для empty/error/degraded состояний

Цель: оператор должен сразу понимать, это «нет данных», «временная задержка», «деградация канала» или «ошибка, требующая действий».

### 6.1 Экран «Сегодня»
- Empty:
  - «За текущую смену пока нет событий. Как только начнутся операции, статистика появится здесь.»
- Slow/degraded:
  - «Обновление данных задерживается. Показываем последний подтверждённый срез от {time}.»
- Error:
  - «Не удалось загрузить оперативную сводку. Проверьте связь с backend и повторите попытку.»

### 6.2 Экран «Краны»
- Empty:
  - «Краны не найдены. Проверьте, добавлены ли устройства в систему и назначены ли точки налива.»
- Controller degraded:
  - «Часть кранов недоступна по связи. Команды управления для них временно заблокированы.»
- Error:
  - «Не удалось получить актуальное состояние кранов. Используйте ‘Повторить’ после восстановления связи.»

### 6.3 Экран «Карты»
- Empty:
  - «Карты и гости пока не зарегистрированы. Добавьте первую карту, чтобы начать обслуживание.»
- Backend degraded:
  - «Связь с backend нестабильна. Операции по картам могут быть отложены до синхронизации.»
- Error:
  - «Не удалось загрузить данные по картам и гостям. Проверьте сеть и повторите запрос.»

### 6.4 Общие принципы текстов
- Всегда называть причину и следующий шаг.
- Не писать абстрактное «Что-то пошло не так» без контекста.
- Для degraded-состояний явно указывать, какие функции всё ещё доступны.
- Для ошибок синхронизации указывать, потеряны ли данные (по умолчанию: «Нет, данные сохранены локально и ждут отправки»).

## 7) Операционные критерии приемки

Считается соответствием NFR, если одновременно выполняются условия:
1. В нормальном режиме 95% 5-минутных окон укладываются в latency SLO (p50/p95).
2. Доля времени в `stale` по критичным статусам < 1% за смену.
3. Fallback с WebSocket на polling происходит автоматически без перезагрузки UI.
4. После reconnect консистентность восстанавливается автоматически не более чем за 5 с для 95% кейсов.
5. Пользователь всегда видит явную индикацию деградации и причины блокировки действий.

## 8) Минимальный набор телеметрии

Для контроля NFR должны отправляться метрики/события:
- `realtime_event_to_ui_latency_ms` (histogram, labels: screen, entity_type);
- `ui_status_staleness_ms` (gauge/histogram, labels: status_type);
- `realtime_transport_mode` (websocket/polling/long_polling);
- `reconnect_attempt_count` и `reconnect_duration_ms`;
- `ui_degraded_mode_active` (boolean + reason);
- `reconciliation_conflict_count`.

Это позволяет оператору и SRE-команде видеть не только факт деградации, но и её первопричину и длительность.
