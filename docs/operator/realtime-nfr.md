# Realtime NFR и UX degraded-режимов

## 1) Целевые метрики realtime

### 1.1 Event-to-UI latency (сквозная задержка от события до отображения)

> Определение: время от фиксации доменного события в backend/controller до момента, когда пользователь видит актуальное состояние в UI.

| Метрика | Цель | SLO (за 5 минут) | SLI источник |
|---|---:|---:|---|
| p50 event-to-UI latency | ≤ 300 мс | 95% интервалов | backend event timestamp + UI render timestamp |
| p95 event-to-UI latency | ≤ 1200 мс | 95% интервалов | backend event timestamp + UI render timestamp |
| p99 event-to-UI latency | ≤ 2500 мс | 99% интервалов | backend event timestamp + UI render timestamp |

Допуски при деградации канала (например, fallback на polling):
- p50 ≤ 800 мс;
- p95 ≤ 3000 мс;
- p99 ≤ 6000 мс.

### 1.2 Максимальная допустимая «устарелость» статусов (status staleness)

> Определение: текущий момент времени минус `updated_at` последнего подтверждённого статуса в UI.

| Категория данных | Допустимая устарелость | Критический порог |
|---|---:|---:|
| Статус налива/дозирования | ≤ 2 с | > 5 с |
| Статус доступности кранов | ≤ 3 с | > 10 с |
| Платёжный/NFC статус | ≤ 2 с | > 5 с |
| Служебные health-статусы (controller/display-agent) | ≤ 5 с | > 15 с |

Правило окраски устарелости:
- `fresh`: до 50% допустимого порога;
- `aging`: 50–100% допустимого порога (жёлтый индикатор);
- `stale`: >100% (красный индикатор + ограничение части действий).

## 2) Realtime transport: websocket + fallback

### 2.1 Приоритет каналов
1. **Primary:** WebSocket (двунаправленный realtime-канал).
2. **Fallback #1:** Short polling (HTTP, интервал 1 с) — если WebSocket недоступен.
3. **Fallback #2:** Long polling / reduced polling (интервал 3–5 с) — при повторных сетевых ошибках.

### 2.2 Правила переключения
- Переход с WebSocket на polling после:
  - 3 подряд ошибок соединения, или
  - отсутствия heartbeat/keepalive > 10 с.
- Возврат с polling на WebSocket:
  - попытка каждые 30 с (экспоненциальный backoff до 2 мин);
  - после успешного handshake выполняется resync snapshot.

### 2.3 Требования к heartbeat и reconnect
- Heartbeat WebSocket: каждые 5 с.
- Таймаут heartbeat: 10 с.
- Reconnect backoff: 1 с → 2 с → 5 с → 10 с → 20 с → 60 с → 120 с (потолок).
- Jitter: ±20% для снижения thundering herd.

## 3) Поведение UI при частичной недоступности компонентов

## 3.1 Недоступен backend

### Блокируется
- Старт/стоп операций, требующих подтверждения backend (налив, подтверждение транзакции, изменение конфигурации).
- Любые действия, изменяющие критичное состояние.

### Доступно с предупреждением
- Просмотр последнего известного состояния.
- Локальные некритичные действия (навигация, открытие экранов справки/логов).

### UX-индикация
- Глобальный banner: «Нет связи с backend. Показаны последние данные.»
- Каждая критичная кнопка disabled + tooltip с причиной.

## 3.2 Недоступен controller

### Блокируется
- Физические операции с краном: старт/продолжение/останов налива.

### Доступно с предупреждением
- Просмотр очереди/сессий и журнала событий.
- Подготовка операции (предвыбор) без фактического запуска.

### UX-индикация
- Статус компонента controller = `offline/degraded` на видимой панели состояния.
- Для tap-card: «Устройство временно недоступно» + кнопка «Повторить».

## 3.3 Недоступен display-agent

### Блокируется
- Функции, требующие синхронного управления внешним экраном/киоском.

### Доступно с предупреждением
- Основной операторский workflow в UI.
- Отложенная отправка команд display-agent (best-effort очередь) при допустимости бизнес-логикой.

### UX-индикация
- Non-blocking предупреждение: «Проблема связи с display-agent. Основные операции доступны.»
- Отдельный счётчик/иконка неприменённых display-команд.

## 4) UX degraded-режимов

### 4.1 Какие действия блокируются
Блокируются действия, которые могут привести к:
- физическому риску (запуск налива без подтверждения контроллера);
- финансовой неконсистентности (подтверждение платежа без backend);
- рассинхронизации критичных состояний (изменение статуса заказа при stale-контексте).

### 4.2 Какие доступны с предупреждением
Допускаются read-only и low-risk операции:
- просмотр карточек, журналов, истории;
- фильтрация/поиск;
- подготовка черновиков действий (без commit).

Все такие действия сопровождаются:
- inline-warning возле данных,
- отметкой времени «Обновлено N сек назад».

### 4.3 Как отображается stale data
Для каждого виджета с live-данными:
- timestamp последнего обновления (`updated_at`);
- цветовой индикатор свежести (fresh/aging/stale);
- при `stale`:
  - красный chip «Данные устарели»;
  - отключение опасных CTA;
  - предложение «Обновить сейчас» (ручной refresh).

Для системного уровня:
- общий health-widget с состоянием backend/controller/display-agent;
- фиксированный banner при деградации более 10 с.

### 4.4 Как восстанавливается консистентность после reconnect
1. Устанавливается транспорт (WebSocket или polling).
2. UI запрашивает **полный snapshot** состояния (authoritative source).
3. Локальные optimistic/queued состояния маркируются как `reconciling`.
4. Выполняется reconciliation:
   - подтверждённые backend события применяются;
   - конфликтные локальные изменения откатываются или переводятся в retry.
5. UI показывает toast: «Связь восстановлена. Данные синхронизированы.»
6. Снимаются временные блокировки действий.

Технические требования на reconciliation:
- идемпотентность операций по `operation_id`;
- дедупликация событий по `event_id`;
- монотонный `version`/`sequence` для порядка применения.

## 5) Операционные критерии приемки

Считается соответствием NFR, если одновременно выполняются условия:
1. В нормальном режиме 95% 5-минутных окон укладываются в latency SLO (p50/p95).
2. Доля времени в `stale` по критичным статусам < 1% за смену.
3. Fallback с WebSocket на polling происходит автоматически без перезагрузки UI.
4. После reconnect консистентность восстанавливается автоматически не более чем за 5 с для 95% кейсов.
5. Пользователь всегда видит явную индикацию деградации и причины блокировки действий.

## 6) Минимальный набор телеметрии

Для контроля NFR должны отправляться метрики/события:
- `realtime_event_to_ui_latency_ms` (histogram, labels: screen, entity_type);
- `ui_status_staleness_ms` (gauge/histogram, labels: status_type);
- `realtime_transport_mode` (websocket/polling/long_polling);
- `reconnect_attempt_count` и `reconnect_duration_ms`;
- `ui_degraded_mode_active` (boolean + reason);
- `reconciliation_conflict_count`.

Это позволяет оператору и SRE-команде видеть не только факт деградации, но и её первопричину и длительность.
