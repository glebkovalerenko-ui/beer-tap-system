# KPI validation: demo и пилот в баре

## Цель документа
Зафиксировать измеримые KPI для операторского контура и задать единые критерии приемки в двух режимах:
- **Demo** (контролируемая среда, «чистые» сценарии);
- **Pilot** (реальная смена в баре, живые данные и шум).

---

## 1) KPI baseline + target

> Все KPI замеряются отдельно для Demo и Pilot.  
> Формат времени: минуты/секунды (P50 и P90), плюс доля попаданий в SLA.

### 1.1 MTTA — Mean Time To Acknowledge
**Определение:** время от появления инцидента в ленте/дашборде до первого осмысленного принятия оператором (acknowledge / assigned / in-progress).

**Baseline (до оптимизаций интерфейса):**
- P50: **2:30**
- P90: **5:00**
- SLA: не менее **70%** инцидентов ack ≤ 3 мин

**Target (после оптимизаций, целевой режим):**
- P50: **1:00**
- P90: **2:00**
- SLA: не менее **90%** инцидентов ack ≤ 2 мин

### 1.2 MTTR — Mean Time To Resolve
**Определение:** время от создания инцидента до перевода в финальный статус resolved/closed (включая коммуникацию и исполнение).

**Baseline (до оптимизаций интерфейса):**
- P50: **18:00**
- P90: **35:00**
- SLA: не менее **65%** инцидентов закрыты ≤ 20 мин

**Target (после оптимизаций, целевой режим):**
- P50: **10:00**
- P90: **20:00**
- SLA: не менее **85%** инцидентов закрыты ≤ 15 мин

### 1.3 False-positive rate инцидентов
**Определение:** доля инцидентов, которые после проверки отмечены как «ложное срабатывание» (не требуют действий в операциях).

Формула:
\[
\text{False-positive rate} = \frac{\#\text{false-positive incidents}}{\#\text{all incidents}} \times 100\%
\]

**Baseline (до калибровки порогов):** **22%**

**Target (после калибровки и UX-улучшений):**
- **Demo:** ≤ **15%**
- **Pilot:** ≤ **12%**

### 1.4 % задач, решаемых без перехода в Settings
**Определение:** доля операторских задач, завершенных из рабочих экранов («Сегодня», «Инциденты», быстрые действия) без навигации в Settings.

Формула:
\[
\text{No-Settings resolution} = \frac{\#\text{tasks completed without Settings}}{\#\text{all operator tasks}} \times 100\%
\]

**Baseline:** **58%**

**Target:**
- **Demo:** ≥ **80%**
- **Pilot:** ≥ **75%**

### 1.5 Время до ответа на 3 ключевых вопроса с экрана «Сегодня»
**Три ключевых вопроса оператора:**
1. Что горит прямо сейчас (критичные инциденты)?
2. Где просадка/риск по выручке или потоку? 
3. Какие действия нужны в ближайшие 30 минут?

**Определение метрики:** время от открытия экрана «Сегодня» до корректного устного/письменного ответа на каждый из 3 вопросов (замеряется модератором).

**Baseline:**
- Q1: **0:45**
- Q2: **1:30**
- Q3: **2:00**
- Суммарно (3 ответа): **4:15**

**Target:**
- Q1: ≤ **0:20**
- Q2: ≤ **0:45**
- Q3: ≤ **0:55**
- Суммарно (3 ответа): ≤ **2:00**

## 2) Сценарный usability-checklist (Demo + Pilot)

Ниже — критические задачи оператора для прогонки на демо и в пилоте. Для каждой задачи задано:
- **Expected time-to-complete (ETC)** — ожидаемое время выполнения;
- **Error budget** — допустимая доля неуспешных прохождений (или критических ошибок).

### Задача 1. Принять новый критичный инцидент
- **Сценарий:** в ленте появляется новый критичный алерт, оператор должен взять его в работу.
- **Критерий успеха:** инцидент переведен в статус acknowledged/in-progress, назначен ответственный.
- **ETC:** ≤ **45 сек**
- **Error budget:** ≤ **5%** критических ошибок (пропуск/неверный инцидент/нет назначения)

### Задача 2. Найти причину и выполнить корректирующее действие
- **Сценарий:** по инциденту нужно открыть карточку, проверить контекст и применить рекомендованное действие.
- **Критерий успеха:** действие выполнено, оставлен комментарий, статус обновлен.
- **ETC:** ≤ **3:00**
- **Error budget:** ≤ **10%**

### Задача 3. Отфильтровать ложное срабатывание
- **Сценарий:** оператор проверяет инцидент и помечает как false-positive с причиной.
- **Критерий успеха:** инцидент закрыт как ложный, причина заполнена корректно.
- **ETC:** ≤ **1:30**
- **Error budget:** ≤ **8%**

### Задача 4. Ответить на 3 ключевых вопроса с экрана «Сегодня»
- **Сценарий:** без перехода в Settings оператор озвучивает ответы на Q1/Q2/Q3.
- **Критерий успеха:** 3 корректных ответа, подтвержденных данными на экране.
- **ETC:** ≤ **2:00** суммарно
- **Error budget:** ≤ **10%** (любая фактическая ошибка в ответе считается ошибкой)

### Задача 5. Завершить инцидент с корректным reason code
- **Сценарий:** после выполненного действия оператор закрывает инцидент и указывает reason code.
- **Критерий успеха:** статус closed/resolved, reason code и итоговый комментарий заполнены.
- **ETC:** ≤ **1:00**
- **Error budget:** ≤ **5%**

### Задача 6. Обработать очередь из 3 инцидентов по приоритету
- **Сценарий:** оператор получает 3 инцидента разных severity и должен обработать в правильном порядке.
- **Критерий успеха:** порядок соответствует приоритету, все 3 инцидента взяты в работу без пропусков.
- **ETC:** ≤ **4:00**
- **Error budget:** ≤ **12%**

### Задача 7. Выполнить типовую операционную задачу без Settings
- **Сценарий:** оператор меняет режим работы/подтверждает действие из рабочего экрана, не открывая Settings.
- **Критерий успеха:** задача завершена, факт действия отражен в журнале.
- **ETC:** ≤ **1:30**
- **Error budget:** ≤ **10%**

## 3) Правила фиксации результата

1. Для каждого участника фиксировать:
   - успешность (success/fail),
   - время выполнения,
   - тип ошибки (навигация, интерпретация, подтверждение, завершение).
2. Разделять результаты по режимам **Demo** и **Pilot**.
3. По итогам каждой волны (не реже 1 раза в неделю пилота) обновлять:
   - baseline/target-геп,
   - топ-3 UX-барьера,
   - план корректирующих изменений.
4. KPI считаются принятыми, если в двух последовательных замерах Pilot выполняются целевые пороги по MTTA, MTTR и no-Settings resolution.
